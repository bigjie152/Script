## 【板块：时间线 Timeline（工程行动轨迹）】
时间线不是故事文本，而是“可核对的行动轨迹表（不用表格，用清单）”。
你的目标：让系统能检测冲突、让 DM 能按图跑局、让审查能定位证据。

### 硬性要求
- 至少覆盖案发前 3 小时到案发后 1 小时（或核心事件窗口）
- 至少 24 个时间点（精确到分钟）
- 每个时间点输出“多条并发事件”（至少 2 条）
- 每条事件必须包含：
  - 时间 / 地点 / 人物 / 行为 / 目的 / 可见性（谁能看到）/ 留下痕迹（可转线索）

### 输出要求
- 只输出 JSON，不要解释
- items 为多条
- content 字段为纯文本（非 Markdown），使用【】标题与分行结构

### 强制输出格式

【Timeline（行动轨迹）】

【时间点 01】
- 时间：
- 并发事件：
  - 事件 A
    - 地点：
    - 人物：
    - 行为：
    - 目的：
    - 可见性（哪些角色可能看到/听到/推断到）：
    - 留下的痕迹（可成为线索的客观痕迹）：
  - 事件 B
    - （同字段）

（至少 24 个时间点）

【冲突自检（必须输出）】
- 可能冲突点（若无也要写“未发现”）：
- 最易被误读的时间点（≥3）：

### Output JSON
{
  "preview": "占位预览",
  "items": [
    {
      "type": "Timeline",
      "title": "[18:00] 节点事件",
      "content": "<纯文本内容，必须严格遵守上述结构>",
      "meta": { "time": "18:00" }
    }
  ]
}
